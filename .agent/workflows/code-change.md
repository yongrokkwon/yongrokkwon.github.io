# 코드 변경 기본 지침

## 필수 절차

1. **수정 전 계획 먼저 공유**

   - 코드를 바로 수정하지 않고, 먼저 어떻게 수정할지 계획을 설명한다
   - 변경할 파일, 변경 내용, 예상 결과를 명시한다

2. **사용자 확인 후 수정**

   - 사용자가 계획을 확인하고 승인한 후에만 실제 코드 수정을 진행한다
   - 사용자가 수정 방향을 바꾸고 싶을 경우 대안을 제시한다

3. **예외 상황**
   - 사용자가 명시적으로 "바로 적용해줘", "수정해줘" 등 즉시 수정을 요청한 경우에만 바로 수정한다
   - 단순 오타 수정이나 이전에 합의된 내용의 적용은 바로 진행해도 된다

## 예시

❌ 잘못된 방식:

```
(사용자 요청 후 바로 코드 수정)
```

✅ 올바른 방식:

```
이렇게 수정하면 될 것 같아요:
- 파일: src/components/hero.js
- 변경: A → B
- 결과: 문단이 자연스럽게 흐르게 됨

이대로 진행할까요?
```

## 수정 완료 후 커밋 제안

코드 수정이 완료되면, 커밋이 필요한지 **주관적으로 판단하여 제안**한다.

### 커밋을 제안하는 경우

- ✅ 기능이 완성된 경우 (하나의 논리적 단위가 끝남)
- ✅ 여러 파일에 걸친 연관된 변경이 완료된 경우
- ✅ 버그 수정이나 중요한 변경이 적용된 경우
- ✅ 사용자가 "이제 됐다", "완료" 등의 만족 표현을 했을 때

### 커밋을 제안하지 않는 경우

- ❌ 아직 작업이 진행 중인 경우 (추가 수정 예정)
- ❌ 단순 탐색이나 확인만 한 경우
- ❌ 임시 테스트 코드인 경우
- ❌ 사용자가 연속으로 여러 요청을 하고 있는 경우

### 제안 형식

```
✅ 폰트 크기 조정이 완료되었습니다.

이 변경을 커밋할까요?
> `git commit -m "style: reduce body font size to 18px for better readability"`
```

### 커밋 메시지 컨벤션

- `feat:` 새로운 기능
- `fix:` 버그 수정
- `style:` 스타일/UI 변경
- `refactor:` 리팩토링
- `docs:` 문서 변경
- `chore:` 기타 잡무
